What a data contract contains

A producer-signed commitment — what they publish, with what quality, under what governance. From today's schema-only pattern to the target 5-clause template, with a concrete example for each. Same template whether data flows through the pipeline or through an MCP.

State 1 · Today template
Schema-only contract — template
Laerdal's current pattern: ODCS schema files. The producer signs what they publish at the field level. Identity is minimal, and the access / compliance / quality clauses are absent.
Producers commit to schema. No Domain Owner, no SLA, no access policy — consumers get field types and that's it.

State 2 · Today example
Content Service — today's example
A real Laerdal data contract following the today's pattern. Composition is rich — seven tables, types, PII tags, server info. The other clauses are mostly absent.
A concrete view of the gap: the producer commits to schema, but no one is accountable for the data product. Consumers have no SLA, no quality guarantee, no clear contact when something breaks.

State 3 · Target template
Full data contract — template
The template every producer's data contract follows. Signed by the producer as a commitment to what they publish. One contract can reference many schemas, and each schema belongs to a Domain and feeds a target data product(s) in the catalog.
The producer signs the full contract — committing to schema AND access, compliance, and quality. Same template whether data flows through the pipeline or an MCP.

State 4 · Target example
HPCPR — target example
The HPCPR team's signed data contract. Three schemas referenced — Session, User, Attempt — each belonging to its own Domain, each feeding one or more data products in the catalog.
Same structure as the template, with real values. One signed contract spans three domains because the producer publishes data that belongs to all three.
