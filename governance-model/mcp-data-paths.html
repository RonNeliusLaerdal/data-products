How data flows to and from AI agents

Two read paths, one action loop. Laerdal's MCP policy and FAQ.

Laerdal's MCP policy · 28.05.2026
Read
Use the data journey by default (Case 1). Direct MCP is allowed for prototyping only (Case 2).
Act
Use a separate Action MCP, owned by the product team.
Always
Every MCP registers in an MCP Registry (e.g. Azure API Center, TBD) and passes the standard MCP review.
Case 1Governed read · default

Read. The platform reads your contract and delivers governance — data quality, policy-as-code for sensitive data, access controls — as part of the self-serviced data platform.

Act. The agent calls a separate Action MCP, owned by the product team and reviewed like any product API.

Loop closes through the data journey. Agent writes land in the source product, then re-enter the catalog through the same contract and pipeline. The data journey sees the consequence.

Case 2Direct MCP · exception

Read. The team exposes data directly via their own MCP, outside the data journey. They demonstrate equivalent controls themselves: sensitive data handling, access rights, data quality.

Act. The same team-managed MCP also handles actions — read and act behind one surface. The team owns the application-side controls (authentication, authorisation, audit, approval flow) on top of the data-side controls.

Loop closes inside the product. The team must observe outcomes themselves; the data journey doesn't see what the agent did. Another reason to migrate to Case 1 once the product reaches production.

Frequently asked

Questions that come up most often from product and engineering teams.

Can I use third-party MCP servers (GitHub, Atlassian, Playwright, Datadog, Azure, DB MCPs, …) for PoCs and tool/agent development?

Yes — PoCs and discovery work fall under Case 2: early-stage, team-managed, transparent. Before you connect any MCP, scope two things:

  • What can it do? What operations the MCP can perform with its access token. Prefer read-only for PoCs; write access (creating PRs, closing tickets, sending messages, running queries against production) needs deliberate scoping.
  • What can it see? Whatever the MCP returns becomes visible to the LLM. If that's production DB rows, internal repos, customer logs, or sensitive Confluence pages, treat the MCP as a sensitive integration regardless of what writes it allows.

What changes when the PoC becomes a production tool?

The same trigger as the data-path policy. Once the tool is relied on in production, it needs governance: sensitive-data handling, access controls, audit, and an approval flow on high-impact writes. Register it in the MCP Registry (e.g. Azure API Center, TBD) and run it through the standard MCP review. If the agent acts on Laerdal systems — writing data, deploying, creating tickets, sending messages — that's production-grade by definition.

Does the Action MCP need its own data contract?

Not in the data-contract sense — the Action MCP is an API, not a data product. It needs the same controls as any production API: authentication, authorisation, audit, and an approval flow on high-impact actions. If the action causes new data to land in a source product, that data flows back through the existing data contract for that product.

Who reviews and approves an MCP for production?

Every productionalized MCP — read or action, internal or external — passes the standard 5-stage MCP review and is registered in the MCP Registry (e.g. Azure API Center, TBD). The Data & AI team can help scope the review and clarify which controls apply to the specific MCP.
